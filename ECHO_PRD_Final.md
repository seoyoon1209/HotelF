# Team E 제품 요구사항 문서(PRD)

| 항목 | 내용 |
|---|---|
| 제품명 | Hoteling |
| 제품 유형 | 호텔 예약 취소 예측 및 운영 의사결정 지원 도구 |

---

## 1. 제품 개요

### 1.1 제품 한 줄 소개
Team E는 호텔 예약 정보를 이용해 예약별 취소 가능성을 예측하고, 호텔 직원이 우선 확인할 예약을 정하도록 돕는 의사결정 지원 도구다.

### 1.2 개발 배경
호텔 예약 취소는 객실 재고 운영, 고객 응대, 재판매 준비 및 예상 매출에 불확실성을 만든다. 예약이 많을수록 직원이 모든 예약을 같은 수준으로 검토하기 어렵고, 실제 취소가 발생한 뒤에는 대응 시간이 부족할 수 있다. Hoteling은 예약 시점에 확보할 수 있는 15개 핵심 정보를 활용해 취소 가능성을 미리 계산하고 직원의 검토 우선순위를 정하는 것을 목표로 한다.

### 1.3 제품 비전
예약 취소 예측을 직원의 업무 흐름에 연결하여, 데이터에 기반한 일관된 검토가 가능하도록 한다. 예측 결과는 자동 의사결정에 사용하지 않으며 최종 판단과 조치는 직원이 담당한다.

---

## 2. 문제 정의

### 2.1 현재 문제
- 호텔 직원은 많은 예약 중 취소 가능성이 높은 예약을 사전에 일관된 기준으로 식별하기 어렵다.
- 취소를 늦게 인지하면 고객 확인과 객실 재판매 준비에 사용할 수 있는 시간이 줄어든다.
- 모든 예약을 동일하게 검토하면 담당자의 시간과 연락 자원이 비효율적으로 사용될 수 있다.

### 2.2 문제 발생 원인
- 예약 취소는 리드타임, 보증금 유형, 시장 세그먼트, 평균 일일 요금, 과거 취소 이력, 숙박 박수, 고객 국가 등 여러 변수가 함께 영향을 미친다.
- 15개 이상의 예약 정보가 여러 필드에 나뉘어 있어 사람이 변수 조합을 동시에 비교하기 어렵다.
- 단일 조건이나 고정 규칙만으로 다양한 예약 패턴을 반영하기 어렵다.

### 2.3 핵심 문제
예약별 취소 가능성을 데이터 기반 확률로 제공하여, 직원이 제한된 시간 안에 우선 검토할 예약을 식별하고 후속 조치를 더 일찍 준비할 수 있도록 해야 한다.

## 3. 목표 및 비목표

### 3.1 제품 목표
- 15개 예약 특성을 기반으로 예약별 취소 확률과 예측 결과를 제공한다.
- 취소 확률이 높은 예약을 우선순위에 따라 조회할 수 있게 한다.
- 예측 결과와 15개 입력 특성 정보를 한 화면에서 확인할 수 있게 한다.
- 예측 결과에 대한 최종 판단과 조치 권한을 직원에게 유지한다.

### 3.2 사용자 목표
- 취소 가능성이 높은 예약을 먼저 확인한다.
- 고객 연락이나 객실 재판매 준비를 더 일찍 시작한다.
- 취소 확률과 전체 15개 예약 특성을 함께 확인한 뒤 조치를 결정한다.
- 예측 결과와 직원 조치 이력을 추적한다.

### 3.3 기술 목표
- 15개 특성 기반 후보 모델(XGBoost, Random Forest, Logistic Regression 등)을 시계열 교차검증을 통해 학습 및 관리한다.
- 학습과 추론에서 동일한 15개 특성 정의와 전처리 규칙을 사용한다.
- F1-Score(주지표)뿐만 아니라 ROC-AUC, Accuracy, Recall을 목표 지표로 관리한다.
- 입력 데이터, 전처리 및 모델 버전을 추적할 수 있게 한다.

### 3.4 Out of Focus
- 예측 결과만으로 예약을 자동 취소하지 않는다.
- 객실 가격이나 환불 정책을 자동 변경하지 않는다.
- 고객에게 자동으로 연락하지 않는다.
- 고객에게 불이익을 주는 자동 의사결정에 사용하지 않는다.
- 현재 범위에서 매출, 객실 점유율 또는 취소율 개선을 보장하지 않는다.

## 4. 목표 사용자

### 4.1 주요 사용자
- 호텔 운영 담당자
- 호텔 수익 관리 담당자
- 예약 및 고객 관리 담당자

### 4.2 사용자 특성
- 여러 예약과 객실 재고를 동시에 관리한다.
- 모델 지표 자체보다 어떤 예약을 먼저 확인해야 하는지에 더 관심이 있다.
- 불필요한 고객 연락과 취소 가능성 누락을 함께 고려해야 한다.
- 예측 결과를 최종 결정이 아닌 참고 정보로 사용할 필요가 있다.

### 4.3 사용자 요구사항
- 예약별 취소 확률과 예측 결과를 빠르게 확인할 수 있어야 한다.
- 취소 확률을 기준으로 예약을 정렬하고 필터링할 수 있어야 한다.
- 예측에 사용된 15개 예약 특성 정보를 결과와 함께 확인할 수 있어야 한다.
- 직원이 예측 결과를 검토하고 조치 여부를 직접 결정할 수 있어야 한다.

## 5. 사용자 시나리오

### 5.1 기본 사용자 흐름
1. 예약 데이터가 업로드된다.
2. 시스템이 AI 모델에 필요한 15개 입력 특성의 존재 여부와 데이터 형식을 검사한다.
3. 시스템이 유효한 데이터를 전처리하고 15개 특성 기반 AI 모델에 전달한다.
4. 모델이 예약별 취소 확률과 예측 클래스를 반환한다.
5. 시스템이 예약을 취소 확률이 높은 순서로 정렬한다.
6. 직원이 예약 상세 정보와 15개 입력 특성을 확인한다.
7. 직원이 고객 확인이나 객실 재판매 준비 여부를 직접 결정한다.
8. 시스템이 예측 결과, 모델 버전 및 직원 조치를 저장한다.

### 5.2 User Stories
- 호텔 운영 담당자로서 취소 확률이 높은 예약을 먼저 확인하여 제한된 근무 시간을 중요한 예약에 집중하고 싶다.
- 수익 관리 담당자로서 기간별 예상 취소 예약 규모를 확인하여 객실 재판매 준비 시점을 판단하고 싶다.
- 예약 담당자로서 취소 확률과 15개 예약 정보를 함께 확인한 뒤 고객에게 연락할 필요가 있는지 직접 결정하고 싶다.
- 관리자로서 어떤 모델 버전이 예측에 사용되었는지 확인하고 싶다.

### 5.3 입력부터 결과 확인까지의 흐름
`예약 데이터 → 15개 특성 검증 → 전처리 → 15개 특성 AI 모델 추론 → 취소 확률 및 예측 클래스 생성 → 우선순위 목록 표시 → 직원 검토 및 조치 기록`

## 6. 기능 요구사항

### FR-1. 예약 취소 예측
- **설명**: 15개 입력 특성을 사용하는 AI 모델로 각 예약의 취소 가능성을 계산한다.
- **입력**: 검증과 전처리를 마친 15개 입력 특성
- **처리**: AI 모델을 통한 이진 분류 추론
- **출력**: 예약별 취소 확률, 예측 클래스, 모델 버전

### FR-2. 우선 검토 예약 목록
- **설명**: 예약을 취소 확률이 높은 순서로 정렬하고 조건별로 필터링한다.
- **입력**: 예약 정보와 취소 확률
- **처리**: 취소 확률, 도착 월, 시장 세그먼트 및 고객 유형 기준 정렬·필터링
- **출력**: 직원용 우선 검토 예약 목록

### FR-3. 예약 상세 조회
- **설명**: 선택한 예약의 예측 결과와 모델 입력에 사용된 15개 특성값을 함께 표시한다.
- **입력**: 예약 식별자, 예측 결과, 15개 입력 특성
- **처리**: 예약 데이터와 예측 결과 조회
- **출력**: 예약 상세 정보, 취소 확률, 예측 클래스, 15개 입력 특성 요약

### FR-4. 예측 이력 관리
- **설명**: 예측 결과와 직원 조치를 저장하고 조회한다.
- **입력**: 예약 식별자, 예측 결과, 예측 시각, 모델 버전, 직원 조치
- **처리**: 권한 확인 후 이력 저장 및 조회
- **출력**: 예약별 예측 및 조치 이력

### FR-5. 결과 내보내기
- **설명**: 예측 결과를 업무용 파일로 내보낸다.
- **입력**: 사용자가 선택한 예약과 조회 조건
- **처리**: 권한 확인, 직접 식별정보 최소화, 파일 생성
- **출력**: CSV 형식의 예측 결과

## 7. AI 및 데이터 요구사항

### 7.1 데이터 범위
- Hotel Booking Demand 공개 데이터셋의 City Hotel 예약 기록을 사용한다.
- 정제 전 City Hotel 데이터는 79,330개 레코드다.
- 정제 절차를 거친 모델링 데이터는 78,669개 레코드다.
- 데이터 기간은 2015년 7월부터 2017년 8월까지다.

### 7.2 목표 변수
| 항목 | 값 |
|---|---|
| 변수명 | `cancellation` (또는 `is_canceled`) |
| 유형 | 범주형 이진 값 |
| 값 0 | 미취소 |
| 값 1 | 취소 |

### 7.3 AI 모델 입력 특성 (15개)

| 번호 | 변수명 | 유형 | 의미 |
|---:|---|---|---|
| 1 | `lead_time` | 수치형 | 예약 시점부터 도착일까지의 기간 |
| 2 | `num_nights_weekend` | 수치형 | 주말 숙박일 수 |
| 3 | `num_nights_week` | 수치형 | 주중 숙박일 수 |
| 4 | `meal_type` | 범주형 | 예약 식사 유형 |
| 5 | `market_segment` | 범주형 | 예약 시장 세그먼트 |
| 6 | `is_repeated_guest` | 범주형 | 재방문 고객 여부 |
| 7 | `num_cancel_prev_resv` | 수치형 | 과거 취소 예약 수 |
| 8 | `num_checkin_prev_resv` | 수치형 | 과거 미취소 또는 체크인 예약 수 |
| 9 | `reserved_room_type` | 범주형 | 예약 객실 유형 |
| 10 | `deposit_type` | 범주형 | 보증금 유형 |
| 11 | `customer_type` | 범주형 | 고객 유형 |
| 12 | `average_daily_rate` | 수치형 | 평균 일일 객실 요금 |
| 13 | `num_guest` | 수치형 | 예약 투숙객 수 |
| 14 | `country_code` | 범주형 | 고객 국가 코드 |
| 15 | `arrival_month` | 수치형 | 도착 월 |

### 7.4 전처리 요구사항
- City Hotel 레코드만 사용한다.
- 성인 투숙객 수가 0인 레코드를 제거한다 (`adults == 0`).
- `distribution_channel` 값이 `Undefined`인 레코드를 제거한다.
- `average_daily_rate`의 비정상적으로 높은 값(`adr >= 400`)을 제외한다.
- 주중과 주말 숙박일 수의 합이 0인 레코드를 제거한다.
- 목표 변수와 15개 입력 변수의 데이터 유형을 학습 스키마와 일치시킨다.

### 7.5 데이터 분할 요구사항
- 시계열 누수(Data Leakage)를 방지하기 위해 **5-Fold 시계열 교차검증(Time-Series Cross-Validation)** 방식을 적용한다.
- 개발 세트(Dev Set)와 최종 테스트 세트(Final Test Set)를 시간 순서에 따라 분할한다.

## 8. 시스템 요구사항

### 8.1 시스템 흐름 (Data Workflow)
`예약 데이터 입력 → 15개 특성 검증 및 전처리 → 15개 특성 AI 모델 추론 → 예측 결과 저장 → 예약 목록 및 상세 조회`

### 8.2 Frontend
- **핵심 기능**:
  - 예약 데이터 업로드
  - 취소 확률 기준 예약 목록 조회
  - 예약 상세 정보 및 15개 입력 특성(Feature) 조회
  - 예측 결과 집계 대시보드 조회
- **배포 및 호스팅**: Render (Frontend Hosting)

### 8.3 Backend
- **핵심 기능**:
  - 사용자 인증 및 권한 확인 (Auth)
  - 15개 입력 데이터 검증 및 전처리
  - AI 모델 추론 요청 처리
  - 예측 결과 및 직원 조치 이력 저장
  - 예약 목록, 상세, 집계 조회 API 제공
- **구현 프레임워크**: Python FastAPI
- **배포 및 호스팅**: Render (Backend Web Server)

### 8.4 AI/ML
- **모델 관리**: 15개 특성을 입력받는 AI 모델 버전 관리
- **운용 방식**:
  - 모델 버전 및 입력 스키마(15개 필드) 버전 동시 관리
- **구현 현황**: 서비스 추론 연동 구현 예정

### 8.5 데이터 저장 (Database)
- **핵심 데이터**: PostgreSQL (Render Hosted PostgreSQL Database Server)
- **저장 대상**: 예약 식별자(ID), 입력 데이터 버전, 15개 입력값, 취소 확률, 예측 클래스, 모델 버전, 예측 시각, 직원 조치 이력

### 8.6 외부 연동 및 배포 환경
- **개발 도구 (Development Tools)**: IntelliJ IDEA, PyCharm, Claude Code
- **배포 플랫폼 (Deployment Platform)**: Render
  - Frontend, Backend Web Server, PostgreSQL Database Server 모두 Render 환경에 호스팅 및 배포
- **외부 연동 범위**:
  - 현재 확인된 외부 API 없음
  - PMS(호텔 자산 관리 시스템) 연동은 현 단계 연동 범위에서 제외

## 9. 비기능 요구사항

### 9.1 성능
- 단일 예약과 배치 예약을 모두 처리할 수 있어야 한다.
- 15개 특성 데이터 추론 시 사용자 요청의 목표 응답 시간을 확정해야 한다.
- 배치 처리에서 성공한 행과 실패한 행을 구분해 반환해야 한다.

### 9.2 안정성
- 필수 15개 특성이 누락된 행이 전체 배치 실패로 이어지지 않도록 오류 행을 분리해야 한다.
- 모델 버전과 전처리 버전을 기록해 예측 결과를 재현할 수 있어야 한다.
- 추론 실패 시 자동 고객 조치가 발생하지 않아야 한다.
- 서비스 장애 시 기존 수작업 절차를 계속 사용할 수 있어야 한다.

### 9.3 사용성
- 취소 확률과 예측 클래스를 명확히 구분해 표시해야 한다.
- 예측 결과가 확정된 사실이 아니라 확률 기반 참고 정보임을 표시해야 한다.
- 예약 목록을 정렬, 검색 및 필터링할 수 있어야 한다.
- 오류 메시지는 15개 특성 중 문제가 발생한 필드와 수정 방법을 포함해야 한다.

### 9.4 확장성 및 유지보수성
- 호텔별 데이터와 모델 버전을 분리할 수 있어야 한다.
- 입력 검증, 전처리, 추론 및 저장 모듈을 독립적으로 분리해야 한다.
- 모델 성능과 데이터 분포 변화를 정기적으로 확인할 수 있어야 한다.

## 11. 성공 지표 및 완료 조건

### 11.1 AI 모델 성능 목표 및 수용 기준 (Key Performance Indicators)

* **주요 평가 지표 (Primary Metric)**: F1-Score
* **보조 평가 지표 (Secondary Metrics)**: ROC-AUC, Accuracy

| 지표 | 최소 수용 기준 (Minimum Threshold) | 목표 성능 (Target Goal) | 측정 및 평가 방법 |
| :--- | :---: | :---: | :--- |
| **F1-Score** | **≥ 0.70** | **≥ 0.75** | 5-Fold 시계열 교차검증 평균 |
| **ROC-AUC** | **≥ 0.85** | **≥ 0.90** | Final Test Set 평가 |
| **Accuracy** | **≥ 0.70** | **≥ 0.75** | 전체 예측 정확도 |

### 11.2 제품 지표
- 직원이 우선 검토할 예약을 찾는 데 걸리는 시간
- 직원의 예측 결과 이해도
- 예측 결과가 검토된 비율
- 불필요한 고객 연락 발생률
- 실제 취소 중 사전에 우선 검토된 예약의 비율

### 11.3 완료 조건
- 15개 입력 특성의 이름과 데이터 유형이 학습 스키마와 서비스 입력 스키마에서 일치한다.
- 전처리 데이터 버전과 데이터 분할 방식이 고정된다.
- AI 모델이 버전 관리되고 추론 요청에 정상 응답한다.
- 입력 검증부터 15개 특성 확인 및 예측 결과 조회까지의 전체 흐름이 동작한다.
- 예측 결과와 모델 버전이 저장된다.
- 목표 사용자를 대상으로 기본 사용자 흐름을 검증한다.
