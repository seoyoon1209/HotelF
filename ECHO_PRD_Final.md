## 1. 제품 개요

### 1.1 제품 한 줄 소개

ECHO는 호텔 예약 정보를 이용해 예약별 취소 가능성을 예측하고, 호텔 직원이 우선 확인할 예약을 정하도록 돕는 의사결정 지원 도구다.

### 1.2 개발 배경

호텔 예약 취소는 객실 재고 운영, 고객 응대, 재판매 준비 및 예상 매출에 불확실성을 만든다. 예약이 많을수록 직원이 모든 예약을 같은 수준으로 검토하기 어렵고, 실제 취소가 발생한 뒤에는 대응 시간이 부족할 수 있다. ECHO는 예약 시점에 확보할 수 있는 정보로 취소 가능성을 미리 계산해 직원의 검토 우선순위를 정하는 것을 목표로 한다.

### 1.3 제품 비전

예약 취소 예측을 직원의 업무 흐름에 연결하여, 데이터에 기반한 일관된 검토가 가능하도록 한다. 예측 결과는 자동 의사결정에 사용하지 않으며 최종 판단과 조치는 직원이 담당한다.

### 1.4 현재 상태

- 호텔 예약 데이터 정제 완료
- Google Cloud AutoML 기준 모델 학습 및 평가 완료
- 입력 특성 5개를 제거한 최종 AutoML 모델 학습 및 평가 완료
- 최종 입력 특성 10개 확정
- 사용자 화면, 추론 API, 운영 데이터베이스 및 서비스 배포는 미구현

## 2. 문제 정의

### 2.1 현재 문제

- 호텔 직원은 많은 예약 중 취소 가능성이 높은 예약을 사전에 일관된 기준으로 식별하기 어렵다.
- 취소를 늦게 인지하면 고객 확인과 객실 재판매 준비에 사용할 수 있는 시간이 줄어든다.
- 모든 예약을 동일하게 검토하면 담당자의 시간과 연락 자원이 비효율적으로 사용될 수 있다.
- 개인 경험에 의존하면 담당자에 따라 검토 우선순위가 달라질 수 있다.

### 2.2 문제 발생 원인

- 예약 취소는 리드타임, 보증금 유형, 시장 세그먼트, 평균 일일 요금, 고객 국가 등 여러 변수가 함께 영향을 미친다.
- 예약 정보가 여러 필드에 나뉘어 있어 사람이 변수 조합을 동시에 비교하기 어렵다.
- 단일 조건이나 고정 규칙만으로 다양한 예약 패턴을 반영하기 어렵다.

### 2.3 기존 방식의 한계

- 수작업 검토는 처리 가능한 예약 수와 판단의 일관성에 한계가 있다.
- 단일 조건 기반 필터는 여러 특성의 결합 효과를 반영하지 못한다.
- 전체 취소율 집계만으로는 개별 예약의 검토 우선순위를 판단하기 어렵다.
- 실제 호텔의 기존 업무 절차와 비교한 사용자 조사는 아직 수행되지 않았다. [확인 필요]

### 2.4 핵심 문제

예약별 취소 가능성을 데이터 기반 확률로 제공하여, 직원이 제한된 시간 안에 우선 검토할 예약을 식별하고 후속 조치를 더 일찍 준비할 수 있도록 해야 한다.

## 3. 목표 및 비목표

### 3.1 제품 목표

- 예약별 취소 확률과 예측 결과를 제공한다.
- 취소 확률이 높은 예약을 우선순위에 따라 조회할 수 있게 한다.
- 예측 결과와 관련 예약 정보를 한 화면에서 확인할 수 있게 한다.
- 예측 결과에 대한 최종 판단과 조치 권한을 직원에게 유지한다.

### 3.2 사용자 목표

- 취소 가능성이 높은 예약을 먼저 확인한다.
- 고객 연락이나 객실 재판매 준비를 더 일찍 시작한다.
- 취소 확률과 예약 정보를 함께 확인한 뒤 조치를 결정한다.
- 예측 결과와 직원 조치 이력을 추적한다.

### 3.3 기술 목표

- Google Cloud AutoML로 학습한 두 모델만 관리한다.
- 기준 모델의 입력 특성 15개와 최종 모델의 입력 특성 10개를 명확히 구분한다.
- 학습과 추론에서 동일한 특성 정의와 전처리 규칙을 사용한다.
- Accuracy뿐 아니라 PR AUC, ROC AUC, Macro F1, 취소 클래스 Recall 및 F1을 함께 관리한다.
- 입력 데이터, 전처리 및 모델 버전을 추적할 수 있게 한다.

### 3.4 비목표

- 예측 결과만으로 예약을 자동 취소하지 않는다.
- 객실 가격이나 환불 정책을 자동 변경하지 않는다.
- 고객에게 자동으로 연락하지 않는다.
- 고객에게 불이익을 주는 자동 의사결정에 사용하지 않는다.
- 현재 범위에서 매출, 객실 점유율 또는 취소율 개선을 보장하지 않는다.

## 4. 목표 사용자

### 4.1 주요 사용자

- 호텔 운영 담당자
- 호텔 수익 관리 담당자
- 예약 및 고객 관리 담당자

### 4.2 사용자 특성

- 여러 예약과 객실 재고를 동시에 관리한다.
- 모델 지표 자체보다 어떤 예약을 먼저 확인해야 하는지에 더 관심이 있다.
- 불필요한 고객 연락과 취소 가능성 누락을 함께 고려해야 한다.
- 예측 결과를 최종 결정이 아닌 참고 정보로 사용할 필요가 있다.

### 4.3 사용자 요구사항

- 예약별 취소 확률과 예측 결과를 빠르게 확인할 수 있어야 한다.
- 취소 확률을 기준으로 예약을 정렬하고 필터링할 수 있어야 한다.
- 예측에 사용된 예약 정보를 결과와 함께 확인할 수 있어야 한다.
- 예측에 사용된 모델 버전을 추적할 수 있어야 한다.
- 직원이 예측 결과를 검토하고 조치 여부를 직접 결정할 수 있어야 한다.

### 4.4 사용자 페인포인트

- 많은 예약의 검토 순서를 수작업으로 정하는 데 시간이 많이 든다.
- 취소 가능성을 늦게 인지하면 대응 선택지가 줄어든다.
- 예측 결과의 의미와 한계를 이해하지 못하면 결과를 과도하게 신뢰하거나 무시할 수 있다.
- 잘못된 예측을 바탕으로 불필요한 고객 접촉이 발생할 수 있다.

## 5. 사용자 시나리오

### 5.1 기본 사용자 흐름

1. 직원이 예약 데이터 파일을 업로드한다.
2. 시스템이 최종 모델에 필요한 10개 입력 특성의 존재 여부와 데이터 형식을 검사한다.
3. 시스템이 유효한 데이터를 전처리하고 최종 AutoML 모델에 전달한다.
4. 모델이 예약별 취소 확률과 예측 클래스를 반환한다.
5. 시스템이 예약을 취소 확률이 높은 순서로 정렬한다.
6. 직원이 예약 상세 정보와 입력 특성을 확인한다.
7. 직원이 고객 확인이나 객실 재판매 준비 여부를 직접 결정한다.
8. 시스템이 예측 결과, 모델 버전 및 직원 조치를 저장한다.

### 5.2 User Stories

- 호텔 운영 담당자로서 취소 확률이 높은 예약을 먼저 확인하여 제한된 근무 시간을 중요한 예약에 집중하고 싶다.
- 수익 관리 담당자로서 기간별 예상 취소 예약 규모를 확인하여 객실 재판매 준비 시점을 판단하고 싶다.
- 예약 담당자로서 취소 확률과 예약 정보를 함께 확인한 뒤 고객에게 연락할 필요가 있는지 직접 결정하고 싶다.
- 관리자로서 어떤 모델 버전이 예측에 사용되었는지 확인하고 싶다.

### 5.3 입력부터 결과 확인까지의 흐름

`예약 데이터 입력 → 입력 검증 → 전처리 → 최종 AutoML 모델 추론 → 취소 확률 및 예측 클래스 생성 → 우선순위 목록 표시 → 직원 검토 및 조치 기록`

## 6. 기능 요구사항

### FR-1. 예약 데이터 입력

- 설명: 사용자가 분석할 예약 데이터를 파일로 입력한다.
- 입력: CSV 형식의 예약 데이터
- 처리: 파일 형식, 인코딩, 헤더 및 행 수 확인
- 출력: 업로드 성공 여부와 처리 대상 행 수
- 우선순위: Must
- 구현 상태: 미구현

### FR-2. 입력 데이터 검증

- 설명: 최종 모델에 필요한 10개 특성의 존재 여부와 형식을 확인한다.
- 입력: 업로드된 예약 데이터
- 처리: 필수 열, 데이터 유형, 결측값, 범주값 및 수치 범위 검사
- 출력: 유효 데이터와 오류 행 목록
- 우선순위: Must
- 구현 상태: 미구현

### FR-3. 예약 취소 예측

- 설명: 최종 AutoML 모델로 각 예약의 취소 가능성을 계산한다.
- 입력: 검증과 전처리를 마친 10개 입력 특성
- 처리: 최종 AutoML 모델을 통한 이진 분류 추론
- 출력: 예약별 취소 확률, 예측 클래스, 모델 버전
- 우선순위: Must
- 구현 상태: 모델 학습·평가 완료, 서비스 추론 연동 미구현

### FR-4. 우선 검토 예약 목록

- 설명: 예약을 취소 확률이 높은 순서로 정렬하고 조건별로 필터링한다.
- 입력: 예약 정보와 취소 확률
- 처리: 취소 확률, 도착 월, 시장 세그먼트 및 고객 유형 기준 정렬·필터링
- 출력: 직원용 우선 검토 예약 목록
- 우선순위: Must
- 구현 상태: 미구현

### FR-5. 예약 상세 조회

- 설명: 선택한 예약의 예측 결과와 최종 모델 입력값을 함께 표시한다.
- 입력: 예약 식별자, 예측 결과, 10개 입력 특성
- 처리: 예약 데이터와 예측 결과 조회
- 출력: 예약 상세 정보, 취소 확률, 예측 클래스, 입력값
- 우선순위: Must
- 구현 상태: 미구현

### FR-6. 운영 현황 조회

- 설명: 기간과 예약 구분별 예측 결과를 집계한다.
- 입력: 예약별 예측 결과
- 처리: 기간, 시장 세그먼트 및 고객 유형별 집계
- 출력: 취소 확률 분포, 우선 검토 예약 수, 기간별 변화
- 우선순위: Should
- 구현 상태: 미구현

### FR-7. 예측 이력 관리

- 설명: 예측 결과와 직원 조치를 저장하고 조회한다.
- 입력: 예약 식별자, 예측 결과, 예측 시각, 모델 버전, 직원 조치
- 처리: 권한 확인 후 이력 저장 및 조회
- 출력: 예약별 예측 및 조치 이력
- 우선순위: Should
- 구현 상태: 미구현

### FR-8. 결과 내보내기

- 설명: 예측 결과를 업무용 파일로 내보낸다.
- 입력: 사용자가 선택한 예약과 조회 조건
- 처리: 권한 확인, 직접 식별정보 최소화, 파일 생성
- 출력: CSV 형식의 예측 결과
- 우선순위: Could
- 구현 상태: 미구현

## 7. AI 및 데이터 요구사항

### 7.1 데이터 범위

- Hotel Booking Demand 공개 데이터셋의 City Hotel 예약 기록을 사용한다.
- 정제 전 City Hotel 데이터는 79,330개 레코드다.
- 현재 채택된 모델링 데이터는 78,669개 레코드다.
- 데이터 기간은 2015년 7월부터 2017년 8월까지다.
- 실제 운영 호텔 데이터는 아직 학습이나 외부 검증에 사용되지 않았다.

### 7.2 목표 변수

| 항목 | 값 |
|---|---|
| 변수명 | `cancellation` |
| 유형 | 범주형 이진 값 |
| 값 0 | 미취소 |
| 값 1 | 취소 |

### 7.3 기준 AutoML 모델 입력 특성

| 번호 | 변수명 | 유형 | 의미 |
|---:|---|---|---|
| 1 | `lead_time` | 수치형 | 예약 시점부터 도착일까지의 기간 |
| 2 | `num_nights_weekend` | 수치형 | 주말 숙박일 수 |
| 3 | `num_nights_week` | 수치형 | 주중 숙박일 수 |
| 4 | `meal_type` | 범주형 | 예약 식사 유형 |
| 5 | `market_segment` | 범주형 | 예약 시장 세그먼트 |
| 6 | `is_repeated_guest` | 범주형 | 재방문 고객 여부 |
| 7 | `num_cancel_prev_resv` | 수치형 | 과거 취소 예약 수 |
| 8 | `num_checkin_prev_resv` | 수치형 | 과거 미취소 또는 체크인 예약 수 |
| 9 | `reserved_room_type` | 범주형 | 예약 객실 유형 |
| 10 | `deposit_type` | 범주형 | 보증금 유형 |
| 11 | `customer_type` | 범주형 | 고객 유형 |
| 12 | `average_daily_rate` | 수치형 | 평균 일일 객실 요금 |
| 13 | `num_guest` | 수치형 | 예약 투숙객 수 |
| 14 | `country_code` | 범주형 | 고객 국가 코드 |
| 15 | `arrival_month` | 수치형 | 도착 월 |

### 7.4 최종 모델에서 제거한 특성

- `num_nights_weekend`
- `num_nights_week`
- `num_cancel_prev_resv`
- `num_checkin_prev_resv`
- `is_repeated_guest`

위 특성은 기준 AutoML 모델의 특성 중요도 하위 항목으로 분류되어 최종 모델에서 제외됐다.

### 7.5 최종 AutoML 모델 입력 특성

| 번호 | 변수명 | 유형 | 의미 |
|---:|---|---|---|
| 1 | `lead_time` | 수치형 | 예약 시점부터 도착일까지의 기간 |
| 2 | `meal_type` | 범주형 | 예약 식사 유형 |
| 3 | `market_segment` | 범주형 | 예약 시장 세그먼트 |
| 4 | `reserved_room_type` | 범주형 | 예약 객실 유형 |
| 5 | `deposit_type` | 범주형 | 보증금 유형 |
| 6 | `customer_type` | 범주형 | 고객 유형 |
| 7 | `average_daily_rate` | 수치형 | 평균 일일 객실 요금 |
| 8 | `num_guest` | 수치형 | 예약 투숙객 수 |
| 9 | `country_code` | 범주형 | 고객 국가 코드 |
| 10 | `arrival_month` | 수치형 | 도착 월 |

최종 모델에서 상대적으로 중요도가 높은 특성은 `deposit_type`, `market_segment`, `lead_time`, `country_code`, `average_daily_rate`다. 모델 수준의 특성 중요도는 개별 예약의 취소 원인을 직접 의미하지 않는다.

### 7.6 입력 스키마 규칙

- 최종 서비스 추론에는 최종 모델의 10개 특성만 전달한다.
- `split` 열은 데이터 분할 제어용이며 모델 입력에 포함하지 않는다.
- `cancellation`은 학습과 평가에만 사용하며 추론 입력에 포함하지 않는다.
- `arrival_month`는 수치형으로 처리한다.
- `num_guest` 생성 공식을 서비스 구현 전에 고정해야 한다. [확인 필요]
- 범주형 허용값과 미확인 범주 처리 규칙을 정의해야 한다. [확인 필요]

### 7.7 전처리 요구사항

- City Hotel 레코드만 사용한다.
- 성인 투숙객 수가 0인 레코드를 제거한다.
- `distribution_channel` 값이 `Undefined`인 레코드를 제거한다.
- `average_daily_rate`의 비정상적으로 높은 값을 제외한다.
- 주중과 주말 숙박일 수의 합이 0인 레코드를 제거한다.
- 목표 변수와 입력 변수의 데이터 유형을 AutoML 학습 스키마와 일치시킨다.
- `average_daily_rate`가 0인 레코드의 처리 규칙을 최종 데이터 버전과 함께 확정해야 한다. [확인 필요]

### 7.8 데이터 분할 요구사항

- 분할 비율, `split` 값의 의미 및 재현 설정을 확정해야 한다. [확인 필요]
- 미래 시점의 예약 정보가 과거 데이터 학습 과정에 포함되지 않는지 검증해야 한다.
